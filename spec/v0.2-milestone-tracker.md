shipped. This is the first *real* forward milestone (the old 0.2/0.3/0.4 ladder
was ceremony, collapsed into v0.1.0 — see [[project-release-plan]]). v0.2 is

### Cluster A — big product surface

- [ ] **Multimodal I/O** — speech-to-text, text-to-speech, image, video, and
      conversational models as first-class whip capabilities. Greenfield (no spec
      yet). Touches: value/type system (new content kinds), effects & capabilities
      (new effect families per modality), provider adapters (each exposes different
      modality endpoints), IFC (flow/redaction of non-text content), and reuses the
      existing content-addressed blob store (images/audio *are* blobs). **Needs a
      large dedicated planning phase.** Long lead; blocks on nothing → plan early.

### Cluster B — harness / tooling surface (shared seams: owned-harness tool
surface, compute-plane container tiers, IFC egress, capability gating)

- [ ] **Native (real-OS) bash — opt-in** — a second bash tier *alongside* bashkit
      (which is and stays the default, DR-0039). Existing home:
      `spec/native-command-tool-tracker.md` (this milestone activates it). Hard
      parts: (a) **cloud** — real fork/exec can't run in a wasm DO isolate, so it
      must route through the Class A/B container-sidecar seam
      (`compute-plane-design-note.md`); (b) **security** — opt-in, capability-gated,
      IFC egress boundary, no detection-evasion posture. Depends on compute-plane
      container enablement (owner/billing-gated).
- [ ] **MCP support** — let whip agents consume external MCP tool servers
      (whip-as-MCP-client) and/or expose whip as an MCP server. Greenfield as a
      feature (MCP appears only as a concept today: Claude-SDK config refs,
      tool-taxonomy mentions). Security-heavy: MCP servers are egress + arbitrary
      tools → capability gating + IFC screening. Shares Cluster B seams with native
      bash → plan the two together.

### Cluster C — inference-quality surface (measurement-first; the three feed each
other — tokenizer behavior and cache dynamics are what the evals measure)

- [~] **LLM-authorability of the language surface** — **designated fix for
      authoring quality (a), NOT fine-tuning** (Jack 2026-07-17). Two lenses:
      - [x] *Tokenization / boundary alignment* — **ANSWERED 2026-07-17, no action.**
        61-file corpus × o200k + cl100k: structural tokens align cleanly (braces
        0.0%, brackets 0.0%, punct 0.0%, multi-char ops 0.4%, keywords 1.4%
        fragmentation); the only straddles are benign `.member` merges. whip is a
        hybrid (braced bodies + brace-less headers/guards) and **the brace-less
        regions carry no alignment penalty** — the premise is falsified. CLOSED
        (Jack signed off; optional Llama/open-tokenizer spot-check waived). SSOT =
        `spec/tokenizer-alignment-note.md`.
      - [ ] *Model priors* (separate, larger, NOT settled by the above) — do models
        emit correct/natural whip? Measurable with the compiler as grader. Only if a
        real gap shows, weigh syntax changes (migration cost). Paired with adoption
        as the long-run lever.
- [~] **Inference-cache optimization** — **SPIKE DONE 2026-07-19**
      (`spec/inference-cache-note.md`). whip already has solid foundational cache
      support (deterministic byte-stable `[tools,system]` prefix; Anthropic
      `cache_control` on system; OpenAI `prompt_cache_key`). Two bounded wins found,
      not a from-scratch build: **G2 (do first)** cache economics are invisible —
      `TurnUsage` discards cache_read/creation tokens, so no hit-rate metric AND
      spend misprices cached input (ties to the spend-cap work); **G1** no moving
      breakpoint on the conversation prefix, so multi-turn agent loops re-process
      the growing conversation uncached every turn (Anthropic allows 4 breakpoints,
      whip uses 1). **G3** cheap byte-stability guard test. Recommend build G2→G1;
      both small/self-contained. Not yet built — awaiting go-ahead.
- [ ] **Tool-call + compaction evals across languages** — internal quality harness:
      does tool-calling and compaction degrade across *natural* languages
      (Spanish/Chinese/…)? *(interpretation to confirm with Jack — natural vs.
      programming languages.)* An *application* of the BUILT improve/evals subsystem
      (ships in v0.1), not new machinery. Consumes tokenizer findings; wants the
      tool surface (native bash / MCP) stable to be worth measuring.

### Cluster E — in-workflow weight optimization (MINIMAL / PARKED)

- [~] **Export the verifier signal for in-workflow optimization** — PARKED, kept
      deliberately minimal (Jack 2026-07-17: "I will come back to it one day").
      **Scope discipline: whip does NOT own the ML cycle.** No trainer, no
      model-hosting, no specialization-scope decision (per-workflow/tenant/base is
      the user's stack's call, not whip's). whip's *only* legitimate contribution is
      IF — and only if — building it into the language yields a training signal a
      generic transcript pipeline can't get: the **verified/typed trajectory + the
      compiler's accept/reject (typed terminals, fail-closed IFC, schema-valid
      coerce, mechanical negatives from type/IFC/coverage/flow diagnostics)** exposed
      as a *utility/export* that an external training stack consumes. Target = (b)
      in-workflow agent behavior (NOT authoring — that's #tokenizers + adoption).
      SDFT (arXiv:2601.19897) is prior art / the reward-free fallback; whip's angle
      is verifier-in-the-loop because the kernel is a cheap reward. **Everything past
      "does whip uniquely inform the signal?" is out of scope until Jack revisits.**
      No design, no build, no sequencing commitment this milestone.

## Proposed sequencing (draft — refine as we pick a starting point)

1. **Build the generic OpenAI provider (#D) first** — foundational, architecturally
   teed-up (registry already open), unblocks broad model reach that #A and #C want.
2. **Research spikes in parallel:** tokenizers (#C) + inference-cache (#C) — cheap,
   background-able, notes not code; de-risk multimodal's cost model.
3. **Open the multimodal (#A) planning phase** — the long pole; start early.
4. **Plan native bash + MCP together (#B)** — most buildable; shared seams;
   build-sequence gated on compute-plane enablement.
5. **Language evals (#C) last of the group** — consumes tokenizer findings + the
   broader model reach from #D; needs the tool surface stable.

(#E is PARKED — not in the build order; revisit on Jack's return.)

# Workflow Encapsulation Boundary — A Non-Interference Design (v2)

**Status: PROPOSED.** Decision-record style. Governing method: *the abstraction is fixed by the non-interference property it must enforce.* Write the property first; the mechanism is whatever minimally enforces it. A **hole** = a violation of the stated property. **Overcaution** = enforcing more than the property requires. This DR states the property, reads the mechanism off it, quantifies how much of the mechanism already exists, and — new in v2 — closes the four holes an adversarial review found the v1 draft was leaving open while correcting three places where v1 over-enforced.

## Executive summary

The workflow-encapsulation boundary is **not a new abstraction.** It is the existing acts-for / two-axis-label / reader-set machinery (Lean-proven: `canAct_iff`, `leak_safe`/`inject_safe`, `integ_is_conf_dual`, `conf_to_public_needs_public_source`) applied to a **new class of governed actions** — the cross-workflow membrane operations — with a workflow **promoted to a first-class principal only where authority must flow through it** (Q1). The abstraction *shape* is right and needs no reshaping; the v1 draft, however, **under-applied its own algebra in four places** and **recommended one unsound coordination mechanism**. v2 fixes both without adding any new primitive: every fix is a further instantiation of a primitive v1 already cites (reader-sets `leaks`, `dominates`/`inject_safe`, the `internal` marker, NMIF-on-selector as shipped in H8, DR-0030 X2 per-field flow signature).

The core code gaps remain as v1 named them: `IrEffectKind::WorkflowInvoke` (parser lib.rs:1336) and the coordination effect kinds `LeaseAcquire`/`CounterConsume`/`LedgerAppend`/`QueueClaim` (parser lib.rs:1348-1350) all fall through the effect match at `ifc.rs:1306` as `_ => {}` — invoke, cancel **and** all coordination ops are invisible to IFC on *both* axes today; `start_child_workflow_instance` (main.rs:25902) attaches no principal and never re-runs `ifc_admission`; `run_notify_effect` (main.rs:23497) resolves a target by runtime instance id and enforces only existence + payload shape.

The four holes v2 closes, each as one primitive re-instantiated:
- **HOLE 1 / coordination (E-COORD):** v1 governed shared coordination on ONE axis (a write gate). The acquire/consume *outcome* (`remaining`, `holders`, the Held-vs-Contended / Ok-vs-Over bit) is a live confidentiality + identity read channel. Govern it two-axis like invoke; and — because an access-control gate provably cannot make two both-authorized workflows non-interfering — recommend **partition** as the fail-closed default, `shared` as an audited opt-in.
- **HOLE 2 / confused deputy (D2b):** B branches on caller-controlled `input` to select which of B's own authorized effects fires. This is byte-for-byte the H8 signal-selector case; extend NMIF-on-selector to the invoke inbound payload.
- **HOLE 5 / instance-id bearer capability (E2-DYN):** the `internal` marker is enforced lexically by static name, but `emit signal to <instance-expr>` resolves the target by runtime id, bypassing the marker. Re-resolve the target's workflow identity at the runtime door and re-consult the marker.
- **HOLE 6 / milestone payload egress (D3'):** `emit milestone` carries a data payload copied verbatim into the observer's fact space with no leak check. Generalize the DR-0030 X2 per-field flow signature from `complete result` to milestone payloads.

Three **overcaution** corrections make the mechanism proportionate from the start: attenuation/per-instance-authority/invoke-seam admission fire only on **delegating edges**; the coordination check fires only on **cross-principal contention** (≥2 distinct workflow types statically naming the resource); same-package non-internal invoke is **admissible by default**.

## 1. The candidate theorems

Two properties, one algebra. Both are stated over principals in the existing acts-for preorder; a **workflow** is a principal `workflow:<pkg>/<name>` (Q1). We distinguish the STATIC encapsulation property (compile-time, lexical/graph — who may name/start/affect whom) from the DYNAMIC integrity+confidentiality property (runtime, no escalation via invoke, no leak via the return/observe paths).

### T-STATIC — Encapsulation (visibility / naming)

> **(E1) No un-authorized membrane act — at the boundary.** For a cross-workflow act `op` (Q2 op set) from `A` to workflow/resource `B`, the door requires an envelope grant `may(A, op, B)` **iff a principal boundary is crossed**: `package_of(A) ≠ package_of(B)` OR `B` is marked `internal`. Same-package, non-internal targets are admissible by default (package co-membership is the implicit grant); the whole program was already admitted once under one envelope by `ifc_admission` (main.rs:26214). At a boundary, absent a grant the act does not compile / is refused fail-closed (`conf_to_public_needs_public_source`, NMIF.lean). *[v2: gated on the package boundary — see §Overcaution 3; v1 fired E1 on every cross-workflow edge.]*
>
> **(E2) Internal workflows are unnameable from outside their boundary.** If `B` is marked `internal` to package `P`, `may(A, invoke, B)` holds only when `A ∈ P` — mirroring `signal_is_internal` (ifc.rs:914), the `whip signal` refusal (main.rs:35559) and the `@tool` export gate (main.rs:16863).
>
> **(E2-DYN) Internal unnameability survives runtime-resolved delivery.** *[v2 — closes HOLE 5.]* For every runtime cross-instance delivery op — `emit signal to <instance-expr>`, notify, invoke-by-handle — from sender instance `A` to a target instance resolved to workflow `B`: if `B` is `internal` to `P`, the delivery is refused unless `pkg(A) = P`, **regardless of how B's instance id was obtained** (by handle, leaked, guessed, forwarded). Possession of an internal target's instance id confers no delivery authority across the boundary. The runtime name (instance id) cannot launder past the lexical marker because the marker is re-resolved from the target's workflow identity at the seam. This is E2's delivery-side companion; the marker and the runtime door were disjoint in v1.
>
> **(E3) Fact isolation is preserved.** No `op` lets `A` read or write `B`'s instance fact space except through the two already-mediated channels: `B`'s declared terminal outputs and declared `emit milestone` projections, re-derived into the observer's own space (main.rs:25850-25898). This clause is *structural* (UNIQUE(instance_id,name,key), instance-scoped reads). **E3 governs fact-space ISOLATION only; it is orthogonal to, and NOT a substitute for, the payload-confidentiality of what those mediated channels carry — that is D3'.** *[v2: the v1 dismissal of milestone observation as "already mediated" conflated isolation with confidentiality; see HOLE 6.]*

Decidable at compile time from lexical facts (package structure, the `internal` marker, the acts-for graph) plus the invoke/emit/coordination graph — same class as `@tool`/leaf/acyclicity (main.rs:37102-37191, parser lib.rs:4139-4227).

### T-DYNAMIC — No escalation via invoke; no leak via return/observe/coordination

> **(D1) Authority attenuates across DELEGATING invoke edges only.** *[v2: gated — see Overcaution 1.]* Define `is_delegating_edge(A→B) = start_grant_present(edge) ∨ package_of(A) ≠ package_of(B) ∨ declassify_on_edge(edge)`. For a delegating edge, `auth(B_child) = declared(B) ∩ effective(A) [∩ start-grant]` and `effective(A) acts-for auth(B_child)` (child ≤ parent). For a **non-delegating** edge (same package, same envelope, no grant, no declassify), `auth(B_child) = declared(B)`, soundly bounded by the shared envelope ceiling already enforced by whole-program admission — no per-edge re-check. `B` can never, *through a delegating call*, exercise authority `A` does not hold.
>
> **(D2) Carriage never fabricates trust.** The authority/label flowing *through* the call is bounded by the meet of `A`'s read vouchers, exactly as `derived_signal_integrity` bounds a receiver by its emitters (ifc.rs:726). An untrusted caller yields an untrusted callee-on-its-behalf; `dominates(carried, sink_integrity)` gates the sink (`inject_safe`, ReaderSets.lean:127).
>
> **(D2b) NMIF-on-invoke-selector — the confused deputy.** *[v2 — closes HOLE 2.]* B's invoke inbound payload (its `input` surface, `IrWorkflowContractKind::Input`) is a caller-controlled READ SOURCE whose integrity is bounded by the meet of the caller's carried authority — **default-low** absent a governance-vouched `invoke:<pkg>/<name> from <Role>` edge (the confidentiality/integrity dual of `signal:<name> from <Role>`, `derived_signal_integrity`). If a `case`/branch discriminant roots in that payload and a selected arm's effect is a crossing (endorse/declassify) OR requires sink-integrity/authority that dominates the selector's carried integrity — i.e. `NOT dominates(selector_int, required_int)`, inject_safe on the *selector* rather than on the data — the arm is REJECTED unless its effective authority is bounded to what the selector carries. This closes selection-based escalation that D1 (meet on carried authority) and D2 (label on the sink a value flows *into*) provably do not: the discriminant does not flow into the sink, it *selects which crossing runs*. This is the exact dual of the shipped H8 signal-selector clause.
>
> **(D3′) The return AND every milestone projection compose on the confidentiality axis.** *[v2: generalized from complete-result-only — closes HOLE 6.]* Every child→observer data projection — B's `complete result` **and** every `emit milestone "n" of Class {…}` payload B projects — is an egress subject to the same per-field reader-set flow signature (DR-0030 X2): each payload field carries the fact-granular reach of the reads feeding it (own-reads of the emitting rule reach every field; direct `when <Fact> as root` roots refine to that fact's producer reach; all other roots fall back to whole-payload reach, fail-closed). The observer may bind it (`after p reaches "n" as m`, or the invoker's terminal return `r`) only where its sink `dominates` that field's reader set (`leak_safe`). A milestone is NOT exempted by E3: E3 preserves fact-space isolation (structural); D3' governs payload confidentiality (value-flow); both are required. **The milestone-REACHED OCCURRENCE itself** (the observer branching on `after p reaches "n"`, a data-dependent 1-bit observe channel — the milestone dual of the coordination Held-vs-Contended discriminant in E-COORD(b)) is governed by the SAME read-source + NMIF-on-selector treatment: a milestone re-derives a `workflow.invoke.reached:<name>` fact (main.rs:25885-25894), which is a labeled read source under the shipped H8 reached-fact machinery, so a branch on it is subject to NMIF-on-selector exactly as a branch on a signal or the coordination outcome. D3' thus governs the milestone *payload* (per-field reader-set), and the reached-fact read-source governs the *occurrence*. Milestone egress is definitionally identical to complete-result egress under `leak_safe`/`projected_reader_set`, differing only in being mid-flight and non-terminal — irrelevant to the confidentiality axis. The full invoke theorem is the CONJUNCTION of the inbound integrity half (D1/D2/D2b) and the outbound confidentiality half (D3').
>
> **(E-COORD) Two-axis coordination governance + partition default.** *[v2 — replaces v1's single-axis H-coord; closes HOLE 1 and resolves open Q2.]* A named coordination resource `X` carries the standard two-axis label `(reader_set(X), integrity(X))`.
> - **(a) INTEGRITY / inbound.** Every `LeaseAcquire`/`CounterConsume`/`LedgerAppend`/`QueueClaim` on `X` requires `dominates(effective(A), integrity(X))` (`inject_safe`) — the write gate v1 already proposed.
> - **(b) CONFIDENTIALITY / outbound.** The acquire/consume OUTCOME — `remaining` (coordination.rs:247, `cap − consumed − amount`, where `consumed` aggregates ALL writers), `holders` (`AcquireOutcome::Contended{holders}`, coordination.rs:22, the OTHER workflows' identity strings), and the Held-vs-Contended / Ok-vs-Over discriminant bit — is a read SOURCE labeled `reader_set(X)`. Every sink `A` routes it to is gated by `leaks(resource:X, sink)`, and any branch on the outcome discriminant is subject to NMIF-on-selector exactly as the shipped H8 signal selector. This makes coordination the SECOND bidirectional membrane op (integrity-inbound ∧ confidentiality-outbound), composed like invoke's D1∧D3' rather than like one-way signals.
> - **(c) NON-INTERFERENCE FORK.** (a)+(b) enforce leak-safety and inject-safety at the SET BOUNDARY — no principal *outside* the label learns or writes X — but they do NOT entail non-interference between two principals **both inside** the label. A shared mutable cell (`CounterRow.consumed`, `Contended{holders}`) is an intrinsic bidirectional channel among co-tenants. Strict cross-workflow-TYPE non-interference (all instances of one workflow type are one principal, keyed `<pkg>/<name>`; NOT cross-INSTANCE — two instances of the SAME workflow still share their type's cell, see §5 "Not closed by any fix") holds **iff coordination rows are PARTITIONED by owner** (`<pkg>/<name>::X`) — the E3 `UNIQUE(instance_id,...)` isolation lifted from the fact store to the coordination store. **Partition is the fail-closed DEFAULT; `shared` is an explicit, audited opt-in** in which the owner knowingly accepts the co-tenant channel. See §5 Decision 1 — this is the one genuine mechanism fork.

**Crucial scope subtlety (from H8, unchanged):** the meet in D1/D2 bounds the authority *carried through* the call, NOT `B`'s ambient authority. `B` still authorizes its own governed reads against its own `declared(B)` ceiling; the intersection only narrows it *for this invocation*. "Child ≤ parent" is about *propagated/effective* authority, not a claim that `declared(B) ⊆ declared(A)`.

**Modelable targets.** T-STATIC = graph/lexical predicate (Maude bite, subworkflow-convergence.maude + std-construct-authorization.maude style). D1 = turn-access-grant.maude never-widen re-parameterized to `{declared(B), effective(A)}`, now conditioned on a `delegating` edge flag. D2/D2b = infoflow-signal-carriage.maude `[carry]`/`[inject]` + selector NMIF with `invoke:<B>` / the invoke inbound payload substituted for `signal:<name>`. D3' = the DR-0030 X2 model (`infoflow-field-signature.maude`) with a milestone-projection rewrite identical to the complete-result egress rule. E-COORD = a new `infoflow-coord-carriage.maude` reusing the signal `[inject]` gate plus an `[outcome-read]` rule, whose **negative bite** shows A's consumption reaches B's observed `remaining` under the shared label (the fork made mechanical) and is No-solution under partition. Lean: no new lattice; add `ReaderSets.lean` lemma that two clearances both `dominates` a label do NOT satisfy NMIF w.r.t. a shared cell (`leak_safe` does not entail NMIF among clearances), plus `same_principal_flow_trivial` (self-coordination ⇒ reflexive dominates, justifying the |P(R)|≤1 skip). D2b/D3'/E2-DYN reuse existing NMIF / `flow_conf_trans` / no-laundering statements source-agnostically.

## 2. Decision points Q1–Q4

### Q1 — Principal granularity

**Recommendation (unchanged from v1): compose (d)+(a)+(b), minted as `kind:address` identities, differing only by what they carry.**
- **`invoke:<pkg>/<name>` and `signal:<name>` as labeled RESOURCE ports** for the membrane ops (the H8 pattern) — what T-STATIC ranges over; covers control ops with no principal promotion.
- **`workflow:<pkg>/<name>` as a first-class PRINCIPAL** *only* to carry `declared(B)`/`effective(A)` for D1's delegating-edge attenuation. Promotion is required exactly where the setuid question lives and nowhere else.
- **`package:<pkg>` as an acts-for ancestor** of its workflows, the export-authority unit, generalizing the @tool gate.

Promote to principal only for the clause that needs it; govern everything else as a resource-label. A named coordination resource is a `resource:<name>` with its own two-axis label (E-COORD), not a per-workflow principal. **Sequencing (see §5 Decision 5):** resource-label-only ships in slice 1 (T-STATIC + E-COORD need no principal promotion); `workflow:<pkg>/<name>` + the per-instance authority slot land in slice 2 alongside the cross-package delegation edge.

### Q2 — Membrane operation set

The membrane the theorem MUST govern = **{invoke/start, cancel (only if promoted), emit-signal-to, acquire/consume/append/claim named coordination resource}**.
- **Control ops** (invoke/cancel/signal-to) via the capability/acts-for layer (already effects with `required_capabilities`).
- **Coordination ops** via the two-axis E-COORD label — governed on BOTH the inbound write (integrity) and the outbound outcome (confidentiality), not just the write. *[v2: v1 governed this single-axis.]*
- **Ops 5–6 (observe milestone/terminal, read facts)** are E3 *preservation* clauses for fact-space isolation — BUT the milestone/terminal PAYLOAD is a D3' egress, not exempt (HOLE 6).

Currently ungoverned and forced by the property: **H-notify** (route rule-driven `emit signal to` through the same guard as the CLI path, *and* re-resolve target identity for E2-DYN — HOLE 5), **E-COORD** (label + both-axis check — HOLE 1). Cancel: govern only if/when it becomes a workflow op.

### Q3 — Static vs dynamic

**Recommendation (unchanged): one abstraction, two enforcement times.** T-STATIC + D2b + D3' are fully compile-time decidable (same class as @tool/leaf/acyclicity). D1's intersection is computable statically once `declared(B)`/`effective(A)` are known, but enforced at the invoke seam at runtime because the target instance is runtime-resolved; the mechanism is the existing `ifc_admission` (main.rs:26214) applied to `start_child_workflow_instance` — and, per Overcaution 1, only on delegating edges. E2-DYN is likewise a runtime door (the target instance-expression is runtime-resolved) computed from lexical facts (the attested `internal` marker re-consulted at the seam). No runtime integrity-state machine.

### Q4 — Authority propagation across invoke (setuid)

**Recommendation — the exact rule, v2-gated:**

> For a **delegating** edge (`start_grant_present ∨ cross-package ∨ declassify`): `auth(B_child) = declared(B) ∩ effective(A) [∩ optional-start-grant]`, child ≤ parent (D1), reusing turn-access-grant's proven never-widen `[keep]`/`[drop]` re-parameterized to `{declared(B), effective(A)}`.
> For a **non-delegating** edge: `auth(B_child) = declared(B)`, no per-edge check — sound because whole-program `ifc_admission` already validated every in-package rule under one envelope.

Reasoning is v1's plus the overcaution correction: applying attenuation to *every* invoke edge rejects safe intra-package library decomposition where A and B share one package + envelope and delegate no authority — that edge crosses no principal boundary and carries no cross-principal influence, so the shared envelope ceiling already covers it. A genuine child-exceeds-caller need remains an explicit `endorse`-style audited envelope hatch, never the default.

## 4. Recommended abstraction shape + minimal mechanism

**Shape:** *workflow-encapsulation = existing IFC, four additions, all proportionate.*
1. **Membrane ops as governed doors.** At the `_ => {}` fall-through (ifc.rs:1306) add arms for `WorkflowInvoke` (E1, gated on the package boundary) and the four coordination kinds (E-COORD: acquire/consume/append/claim push a WRITE for the `dominates`/inject check AND register `resource:X` as a READ source for the outcome). Route rule-driven `emit signal to` through the CLI guard and re-resolve target identity at `run_notify_effect` (H-notify + E2-DYN).
2. **`internal` visibility markers.** `internal_workflows` set (sibling of `internal_signals`), declared in the envelope, part of the attested hash, refused fail-closed for out-of-boundary callers (E2) — re-consulted at the runtime delivery door by target-identity re-resolution (E2-DYN).
3. **Attenuated-intersection at the invoke seam — delegating edges only.** A per-instance authority slot set by `start_child_workflow_instance` to `declared(B) ∩ effective(A)` and enforced by an `ifc_admission` call on the child path, fired ONLY when `is_delegating_edge` holds; non-delegating edges keep today's no-principal path (slice 2).
4. **Per-field flow signature on all projections.** `milestone_field_reads` re-seeds `reach_reads_from` so `emit milestone` payloads carry the same DR-0030 X2 signature as `complete result`; egress gated by `leak_safe` at compile time (D3').

**Explicitly NOT doing (overcaution guardrails):**
- **NOT** a new principal *type*/enum — principals stay strings.
- **NOT** promoting every workflow to a principal — only enough for D1's delegating edges (Q1).
- **NOT** a runtime integrity-state machine — D1/D2b/D3' are statically computed; runtime-enforced only at the invoke and delivery seams (Q3).
- **NOT** attenuating same-package non-delegating invoke, running the coordination check on single-principal (self-coordination) names, or requiring a grant for same-package non-internal invoke — all three would reject safe programs (§Overcaution 1-3).
- **NOT** a producer authority attestation across packages — the consumer recompiles pinned source under its OWN envelope (DR-0029/H8), for D1, D2b, D3' and E-COORD alike.
- **NOT** a new declassify/endorse grammar — child-exceeds-parent stays an envelope-level audited hatch.

### Overcaution corrections (baked in)

1. **Attenuate only on delegation.** `is_delegating_edge(A→B) = start_grant_present ∨ package_of(A)≠package_of(B) ∨ declassify_on_edge`. D1's attenuation, the per-instance authority slot, and invoke-seam `ifc_admission` fire only on delegating edges; a same-package/same-envelope/no-grant invoke keeps the current no-principal path, sound under the once-admitted whole-program envelope.
2. **Coordination check only on cross-principal contention.** Let `P(R) = {workflow-principal w : some rule of w statically acquires/consumes R}`. The E-COORD `dominates` gate fires iff `|P(R)| ≥ 2` (distinct workflow types); `|P(R)| ≤ 1` is self-coordination (any number of instances of one workflow) and admits unlabeled — the counterparty is the same principal (Lean `same_principal_flow_trivial`). Note this narrows the *firing condition* of the gate; the partition/shared *default* (Decision 1) is a separate, orthogonal choice for the `|P(R)|≥2` case.
3. **Same-package invoke admissible by default.** E1's door consults `may(A,invoke,B)` only at the package boundary (`package_of(A)≠package_of(B)` OR `B` internal); same-package non-internal targets are admissible (package co-membership is the implicit grant), making E1 fire exactly where E2 does — one membrane definition, two axes. *(Coupled to Decision 2: if in-package defaults to `internal`, in-package invokes may need grants.)*

## 5. Adversarial review (v1 → v2)

An adversarial review confirmed the **abstraction shape is right** — existing acts-for / reader-set / integrity IFC applied to membrane ops, two axes (confidentiality via reader-sets/leaks, integrity via dominates/carriage), workflow promoted to a principal only where authority must flow through it; no reshaping needed. It found the v1 draft **under-applied its own algebra in four places** and **recommended one unsound coordination mechanism**. Each finding and its v2 closure:

- **HOLE 1 — coordination governed single-axis (unsound recommendation).** v1's H-coord attached a label and a `dominates` gate on acquire/consume — governing only naming/writing (integrity/inbound). It left the RETURN VALUES ungoverned: `ConsumeOutcome{remaining}` (coordination.rs:247, over the shared aggregate of all writers) and `AcquireOutcome::Contended{holders}` (coordination.rs:22, other workflows' identity strings) are reads of one workflow's writes by another — a live confidentiality channel plus an overt identity/contention side-channel. Both fall through the same `_ => {}` (ifc.rs:1306) as invoke. v1 also left the label-vs-partition choice "not derivable from the property," but it IS derivable on soundness grounds. **Closed by E-COORD:** two-axis governance (write-gate + outcome-as-`reader_set(X)`-read + NMIF-on-discriminant), and — because an access-control gate provably cannot make two both-authorized workflows non-interfering — **partition recommended as the fail-closed default, `shared` an audited opt-in** (Decision 1).

- **HOLE 2 — confused deputy via invoke payload selector.** B's caller-controlled `input` selects which of B's own individually-authorized effects fires (`case input.mode { "drain" => <privileged effect> }`). D1's meet bounds only the authority carried *through* the call (B's effects stay authorized under ambient `declared(B)`, the §33 subtlety); D2's data-label only gates a sink a value flows *into*, not which of two authorized branches runs. This is byte-for-byte the H8 signal case: the discriminant does not flow into the sink, it selects the crossing. **Closed by D2b:** the invoke inbound payload is a low-integrity read source (default-low absent a vouched `invoke:<pkg>/<name> from <Role>` edge); the shipped NMIF-on-selector check (ifc.rs:1626-1682) is extended to it, broadening the crossing predicate to any arm whose sink-integrity dominates the selector's carried integrity.

- **HOLE 5 — instance-id bearer capability.** E2 unnameability is enforced lexically by static workflow name, but `emit signal to <instance-expr>` / notify resolves its target by RUNTIME instance id in `run_notify_effect` (main.rs:23497), enforcing only `target_exists` + payload shape. A leaked/guessed instance id of an internal workflow reaches it directly, bypassing the marker; same gap on any invoke-by-handle path. The marker and the runtime door were disjoint. **Closed by E2-DYN:** re-derive the target instance's workflow identity (instance→`version_id`→`program_name`) at the runtime door and re-consult the attested `internal_workflows` marker; refuse fail-closed unless the sender's package (derived the same way) matches — the `whip signal` no-laundering refusal generalized from a static name to a runtime-resolved target.

- **HOLE 6 — milestone payload egress un-flow-checked.** `emit milestone "n" of Class { fields }` carries a data payload copied VERBATIM into the parent's `workflow.invoke.reached:<name>` fact (main.rs:25885-25894) and bound via `after p reaches "n" as m` — the exact child→observer confidentiality channel D3/DR-0030 X2 governs for `complete result`, but `result_field_dependency_reads` seeds ONLY from `complete_field_reads`; no `milestone_field_reads` exists. v1's §2 op-5 / E3 dismissed milestone observation as "already mediated," conflating fact-space ISOLATION (E3, structural) with payload CONFIDENTIALITY (value-flow). **Closed by D3':** a `milestone_field_reads` metadata map re-seeds the existing `reach_reads_from`, giving milestone payloads the same per-field reader-set signature as `complete result`; egress gated by `leak_safe`.

**Overcaution found and corrected (v1 → v2):** v1 read its own algebra as if every invoke edge and every shared name were a trust-crossing delegation — it would (1) attenuate + add a per-instance slot + re-run `ifc_admission` on EVERY invoke edge; (2) run label+dominates on ANY contended coordination name; (3) fail-closed "does not compile" for same-package invoke. Applied literally these reject SAFE intra-package library decomposition. v2 introduces one lexical predicate (`is_delegating_edge`) and the `|P(R)|≥2` contention test, gating all three heavy mechanisms so the property is enforced exactly where a principal boundary is actually crossed (§Overcaution 1-3).

## 6. Open decisions (owner-only)

Each is the owner's to decide; recommendation given.

1. **The coordination fork — partition vs label (`shared`). RESOLVED 2026-07-01 (Jack): PARTITION** (fail-closed default; `shared` an explicit, audited, per-resource opt-in). This is the one genuine mechanism fork, and it is *irreducible*. **Partition** (rows keyed `<pkg>/<name>::X`, distinct cells per owner): sound — delivers strict cross-workflow non-interference (E3 isolation lifted to the coordination store), no runtime check needed — but SEMANTICALLY ELIMINATES cross-workflow coordination: a shared rate-limiter / cross-workflow mutex / shared ledger becomes per-workflow-private state (equivalent to run-store local state). **Label + two-axis check (`shared`)**: preserves real cross-workflow coordination and keeps the SET boundary leak- and inject-safe, BUT provably cannot make two co-tenants non-interfering — even with `holders`/`remaining` redacted, the 1-bit Held-vs-Contended / Ok-vs-Over control-flow discriminant remains an adversarially-usable covert channel (and `holders` leaks caller identity). An access-control gate cannot make two both-authorized principals non-interfering; the owner cannot have both cross-workflow coordination AND non-interference. **RECOMMEND: partition as the fail-closed DEFAULT; `shared` an explicit, audited, per-resource opt-in that declares acceptance of the co-tenant channel.** (Part 1's outbound `leaks` check is still required under `shared` to keep the set boundary leak-safe; it does not close the co-tenant channel.)

2. **In-package default visibility. RESOLVED 2026-07-01 (Jack): invocable-in-package by default + an explicit `private` marker to opt a workflow OUT of invocability from outside its boundary.** Cross-package stays closed-by-default like @tool; in-package open-by-default. Default-private would be language-level overcaution (the package boundary already IS the fail-closed membrane) and would force migration. *Naming note:* the marker is `private` on a workflow (Jack's choice); the underlying mechanism is the `internal_workflows` attested marker paralleling the shipped `internal_signals` — consider whether to align the surface keyword (`internal` for both) or keep `private` for workflows / `internal` for signals. (Couples to Overcaution 3.)

3. **Explicit start-grant surface. RESOLVED 2026-07-01 (Jack): `invoke B with access to {…}` ships in v1.** Motivation is primarily COMPOSABILITY / separation-of-concerns (give a subagent exactly the authority it needs and nothing more), not primarily data-security. Semantics are **purely additive**: default (no clause) = the automatic cap `declared(B) ∩ effective(A)`; the `with access to {…}` clause narrows FURTHER, and the grant it names must be ⊆ `declared(B) ∩ effective(A)` (a grant can never widen — naming an authority the parent lacks or the child never declared is an error, the never-widen rule). Reuses the turn-access-grant shape (`authority = profile ∩ grant`) at the invoke seam. Because it requires an authority slot to narrow, it forces Decision 5 to v1 (see below).

4. **Cross-workflow cancel as a language op. RESOLVED 2026-07-01 (Jack): operator-only for v1; DEFER language-level cancel but PLAN a careful dedicated design pass for it + the supervisor pattern** (not to be added without that pass). When promoted it must be a directed door gated by parent-`dominates`-child integrity, and the design pass must address the cancel-timing covert channel (a supervisor observing WHEN a child cancels is a data-dependent observe channel needing the same NMIF-on-selector treatment as the coordination Held/Contended and milestone-reached discriminants). Track as its own DR/tracker item.

5. **Q1 promotion scope. RESOLVED 2026-07-01 (Jack): promote the workflow to a `workflow:<pkg>/<name>` principal + per-instance authority slot in v1** (forced by Decision 3 — `with access to` needs an authority slot to narrow). This pulls the dynamic authority half (D1 automatic cap `declared(B) ∩ effective(A)` + the explicit `with access to` grant) INTO v1; there is no longer a slice-1/slice-2 split — the full theorem (static membrane + dynamic authority + coordination partition) is the v1 scope. The invoke-seam admission (`ifc_admission` lifted to `start_child_workflow_instance`) and the per-instance authority slot are the largest new runtime pieces and should be built early, not deferred. *(Historical option, now rejected: resource-label-only first — an `invoke:<pkg>/<name>` port covering E1/E2/E2-DYN + control ops via the H8 pattern with no per-instance slot, deferring attenuation to a later slice. Rejected because Decision 3 requires the authority slot in v1, so the promotion cannot be deferred; the label-only port is subsumed by the principal.)*

**Not closed by any fix:** `is_delegating_edge` and the E1 boundary predicate assume `package_of` is well-defined for every workflow (holds today). If the open one-program-many-workflows scoping keystone blurs package identity, the boundary predicate must be re-derived. Door discipline is per-seam: any FUTURE runtime op that delivers to / mutates a target instance by id must route through the E2-DYN marker check (the cancel-cleanup terminal-path audit lesson). Under dev/ungoverned envelope the `internal_workflows` loader returns false, so encapsulation is enforced only under a verified envelope — consistent with all IFC.

**Not closed by admission alone (claim-vs-enforcement):** the per-instance authority slot + invoke-seam admission bind `with access to` at the governed-effect seams, but they do NOT bind the owned-harness agent loop's in-turn tool calls — today the live tool executor reads policy from process env, not the declared authority (build tracker Phase 4b closes this). Until that piece lands, a "runtime least-privilege subagent" claim for owned-harness turns would be a report-vs-check gap: the grant governs admission and effect gating, not which tool call runs mid-turn. Docs and the guarantee report must scope the claim accordingly (tracker § Sequencing).
